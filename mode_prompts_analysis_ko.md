# 모드 프롬프트 분석 (Mode Prompts Analysis)

이 문서는 **Architect(설계자)**, **Debug(디버그)**, **Ask(질문)** 세 가지 모드 프롬프트에 대한 비교 분석을 제공합니다. 각 모드는 AI 에이전트의 역할, 권한, 작업 흐름을 정의하여 개발 수명 주기의 각 단계에 맞는 맞춤형 동작을 보장합니다.

## 비교 개요 (Comparative Overview)

| 기능 | Architect 모드 (설계자) | Debug 모드 (디버그) | Ask 모드 (질문) |
| :--- | :--- | :--- | :--- |
| **파일명** | `architect.md.resolved` | `debug.md.resolved` | `ask.md.resolved` |
| **역할** | 기술 리더 / 기획자 | 전문 소프트웨어 디버거 | 기술 어시스턴트 |
| **주요 목표** | 계획 수립, 맥락 파악, 할 일 목록 작성 | 체계적인 문제 진단 및 해결 | 질문 답변, 개념 설명 |
| **편집 권한** | **제한됨**: Markdown (`.md`) 파일만 가능 | **허용됨**: 모든 파일 (단, **로깅 엄격 금지**) | **금지됨**: 어떤 파일도 편집 불가 |
| **필수 도구** | `update_todo_list` (또는 마크다운 작성), `switch_mode` | `dap-proxy`, `macos-control` | 지정되지 않음 |
| **핵심 제약** | 소요 시간 추정(Estimate) 제공 금지 | **"로깅은 디버깅이 아니다"** | 구현 모드로 자동 전환 금지 |

## 상세 분석 (Detailed Analysis)

### 1. Architect 모드 (`architect.md.resolved`)
**초점:** 기획 및 설계 (Planning and Design)
*   **페르소나:** "Roo", 호기심 많고 계획성이 뛰어난 기술 리더.
*   **워크플로우:**
    1.  정보 및 맥락 수집.
    2.  명확화를 위한 역질문.
    3.  구체적이고 실행 가능한 **할 일 목록(Todo List)** (또는 `plan.md`) 작성.
    4.  **중요 단계:** 구현을 시작하기 *전에* 사용자의 승인을 받음.
    5.  `switch_mode`를 사용하여 구현 담당자에게 작업 이관.
*   **고유 제약 사항:** 노력/시간 추정치(Level of Effort) 제공을 명시적으로 금지하며, 오직 기술적인 작업 분해에만 집중합니다. 유일하게 Markdown 파일만 편집할 수 있도록 제한되어 있어, "기획"과 "코딩" 단계의 분리를 강제합니다.

### 2. Debug 모드 (`debug.md.resolved`)
**초점:** 체계적인 진단 및 수리 (Systematic Diagnosis and Repair)
*   **페르소나:** "Roo", 전문 도구를 사용하는 숙련된 디버거.
*   **워크플로우:**
    1.  **환경 점검:** 즉시 `dap-proxy`와 `macos-control` 도구의 활성화 여부를 확인합니다. 누락시 **중단**하고 사용자에게 설치를 안내합니다.
    2.  **전략 수립:** 중단점(breakpoint)과 변수 검사를 활용한 계획 수립.
    3.  **실행:** 런타임 데이터를 기반으로 문제 해결.
*   **고유 제약 사항:** 가장 강력한 도구 사용 요구사항을 가집니다. "console.log"를 이용한 디버깅을 엄격히 금지하며, 지정된 MCP 도구(`dap-proxy`, `macos-control`) 사용을 강제하여 높은 수준의 전문적인 디버깅을 수행하도록 합니다.

### 3. Ask 모드 (`ask.md.resolved`)
**초점:** 교육 및 설명 (Education and Explanation)
*   **페르소나:** "Roo", 지식 있는 기술 어시스턴트.
*   **워크플로우:**
    1.  코드 및 외부 리소스 분석.
    2.  개념 설명 및 질문 답변.
    3.  이해를 돕기 위한 Mermaid 다이어그램 활용.
*   **고유 제약 사항:** 읽기 전용 모드입니다. 편집이 **금지**되어 있습니다. 이는 사용자가 의도치 않은 코드 변경에 대한 걱정 없이 질문할 수 있는 안전한 "상담" 계층을 제공합니다.

## 결론 (Conclusion)

세 가지 모드는 구조화된 개발 파이프라인을 형성합니다:
1.  **Ask**: 시스템 이해.
2.  **Architect**: 변경 계획 수립.
3.  **Debug**: 발생한 복잡한 문제 해결 (일반적인 구현과 구분됨).

각 모드는 특정 작업에 기이전트의 행동을 최적화하기 위해 권한과 필수 도구라는 "가드레일"을 가지고 있습니다.
